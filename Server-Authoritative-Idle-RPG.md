# 📑 คู่มือเจาะลึกโครงสร้างระบบ: Server-Authoritative Idle RPG

เอกสารฉบับนี้จะพาคุณไปสำรวจว่าแต่ละส่วนทำงานอย่างไร และทำไมเราถึงออกแบบเช่นนั้น เพื่อเป็นพื้นฐานในการพัฒนาฟีเจอร์ระดับสูงต่อไป

---

## 1. ปรัชญาการออกแบบ: Server-Authoritative
ในเกมทั่วไป ผู้เล่นอาจโกงได้โดยการแก้ค่า HP หรือ Gold ในเครื่องตัวเอง แต่ในระบบนี้:
*   **Frontend (Next.js):** เป็นเพียง "จอแสดงผล" และ "เครื่องจำลองภาพ" (Visual Simulator) เท่านั้น เพื่อให้ผู้เล่นรู้สึกถึงความต่อเนื่องลื่นไหล
*   **Backend (Golang):** คือ "ผู้ตัดสิน" (The Arbitrator) ทุกการกระทำต้องผ่านการคำนวณและยืนยันจาก Go Server ก่อนจะถูกบันทึกลงฐานข้อมูล ข้อมูลจาก Client จะถูกตรวจสอบอย่างเคร่งครัด

## 3. เจาะลึก Logic สำคัญ (Code Deep Dive)

### A. ระบบต่อสู้และการคำนวณ Damage (`engine/combat`)
*   **Attribute Scaling:** เราไม่ได้ใช้ตัวเลขดิบๆ แต่เราใช้ระบบ **Derived Stats** (ค่าพลังที่สกัดออกมา) เช่น Strength 1 แต้ม จะกลายเป็นพลังโจมตี 3 หน่วย ระบบนี้ช่วยให้คุณปรับสมดุลเกม (Balance) ได้ง่ายในอนาคตที่ไฟล์ `combat.go`
*   **Critical Hit:** มีระบบสุ่มโอกาสติดคริติคอล (200% Damage) โดยคำนวณจากค่า Agility
*   **Elemental Advantage:** มีการใช้ตัวคูณ 1.5x (ชนะทาง) และ 0.8x (แพ้ทาง) ตามความสัมพันธ์ Fire > Wood > Water > Fire

### B. ระบบการเติบโตขณะออฟไลน์ (`engine/progression`)
*   **TTK (Time to Kill):** นี่คือสูตรลับของเกม Idle เราคำนวณว่า "มอนสเตอร์ 1 ตัว ผู้เล่นใช้เวลากี่วินาทีในการฆ่า" จากนั้นเอาเวลาที่ผู้เล่นหายไปมาหาร เพื่อหาจำนวนมอนสเตอร์ที่ตายจริง
*   **Weighted Loot Table:** ระบบสุ่มของที่ใช้ "น้ำหนัก" (Weight) แทนเปอร์เซ็นต์ดิบๆ เพื่อให้คุณปรับอัตราดรอปของระดับ Legendary ให้ยากระดับ 0.01% (น้ำหนัก 1 จาก 10,000) ได้อย่างแม่นยำ
*   **Level Up Scaling:** ใช้สูตร Exponential `BaseExp * (Level ^ 1.5)` เพื่อให้การเลเวลอัปท้าทายขึ้นเรื่อยๆ

### C. ระบบฐานข้อมูลและการคงอยู่ของข้อมูล (`engine/database`)
*   **Persistence:** ข้อมูลตัวละคร (Stats, Level, Gold, LastSync) และไอเทม จะถูกเก็บใน PostgreSQL
*   **Upsert Pattern:** ในไฟล์ `repository.go` เราใช้ `ON CONFLICT (id) DO UPDATE` เพื่อให้การบันทึกข้อมูลทำได้ในคำสั่งเดียว ไม่ว่าจะสร้างใหม่หรืออัปเดตของเดิม

## 4. วงจรการทำงานของข้อมูล (Data Lifecycle)

เมื่อคุณรันระบบ วงจรจะทำงานดังนี้:

1.  **Sync Trigger:** Next.js ส่งคำขอ Sync พร้อม `CharacterID` ไปที่ Backend
2.  **Load State:** Go Server ดึงข้อมูลล่าสุดจาก **PostgreSQL** (เช่น เลเวลล่าสุด, เวลาที่ Sync ครั้งสุดท้าย)
3.  **Process Offline:** Go คำนวณเวลาที่หายไป -> คำนวณมอนสเตอร์ที่ตาย -> คำนวณ EXP/Gold/Loot
4.  **Level Up Check:** ระบบตรวจสอบว่า EXP ที่ได้ใหม่ ทำให้เลเวลอัปกี่ครั้ง และอัปค่าพลัง (Str, Agi, Vit) ให้ทันที
5.  **Persist:** บันทึกค่าใหม่ทั้งหมดกลับลง **PostgreSQL** เพื่อยืนยันสถานะล่าสุด
6.  **Response:** ส่งสถานะใหม่ล่าสุดกลับไปที่ **Next.js** เพื่อบังคับอัปเดตหน้าจอผู้เล่น (Override Local State)

## 5. คำแนะนำในการศึกษาต่อ

หากคุณต้องการแก้ไขหรือเพิ่มฟีเจอร์ ให้เริ่มดูที่ไฟล์เหล่านี้ตามลำดับ:

1.  **ถ้าอยากปรับความแรง/ความอึด:** ไปที่ `engine/combat/combat.go`
2.  **ถ้าอยากปรับให้อัตราดรอปของยากขึ้น:** ไปที่ `engine/progression/loot_table.go`
3.  **ถ้าอยากเปลี่ยนสูตร EXP:** ไปที่ `engine/progression/leveling.go`
4.  **ถ้าอยากเพิ่มคอลัมน์ในฐานข้อมูล:** ไปที่ `engine/database/db.go` (อย่าลืมอัปเดต Struct ใน `repository.go` ด้วย)
5.  **ถ้าอยากปรับ UI บนเว็บ:** ไปที่ `components/GameDashboard.tsx`
